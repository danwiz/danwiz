# Contributing

Thank you for helping build practical, transparent technology-readiness tools.

## Contribution workflow

1. Start from an issue describing the user problem and intended outcome.
2. Keep changes small, documented, and testable.
3. Update schemas, examples, and version notes together when behavior changes.
4. Do not add private, client, credential, or production data.
5. Explain scoring changes and preserve deterministic calculations.
6. Include accessibility and security considerations in the pull request.

## Definition of done

A contribution is ready when its purpose is clear, documentation is current, sample data is synthetic, validation passes, and no unresolved governance or security concern remains.

## Governance

New readiness domains, maturity levels, scoring semantics, or evidence classes require an architecture decision and impact review before adoption.
